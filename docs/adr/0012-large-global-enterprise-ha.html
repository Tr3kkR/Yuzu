Context

Yuzu's first enterprise HA design must be sized for a large global enterprise: roughly 2,200 locations across 56 countries, 1.2 million active devices, and 210,000 staff. This is not a single-region availability problem. Multiple warm regions are required from the first implementation.

Endpoint management is operational infrastructure at this scale. Yuzu will be the control plane through which humans and Agentic Colleagues query, command, scan, patch, quarantine, enforce policy, and observe endpoint posture. Outage time becomes manual triage, duplicated regional work, spreadsheet chasing, delayed remediation, and slower incident containment.

Commercial endpoint platforms already train enterprises to expect cloud-style availability, service health, documented SLOs, and supportable disaster recovery. Yuzu's opportunity is to offer that posture in a self-hostable and inspectable form, with a low-latency command path designed for Agentic Colleagues rather than a black-box SaaS control plane.

Decision

Decision: Yuzu will implement a large-global-enterprise HA profile as the first HA target. The primary shape is macro-regional cells across EMEA, APAC, and Americas, each with active-active gateways, horizontally replicated servers, HA Postgres, regional data residency, and at least one manually promoted warm standby region.

Gateway Layer

Gateways own live streams only. They publish health, load, drain state, build version, active stream count, queue depth, BEAM mailbox pressure, memory pressure, and upstream health. Agents receive a signed gateway directory and reconnect using sticky selection, latency, load, health, and jittered backoff.

Durable Presence

Regional Postgres stores gateway membership and agent presence. Agent ownership is fenced with connection_epoch, gateway_id, and session_id so stale gateways cannot overwrite newer stream ownership.

Durable Command Mailbox

Accepted commands, target rows, delivery leases, attempts, and response sequence keys are stored in Postgres before delivery. Gateways claim target rows for connected agents; gateway death is recovered by lease expiry and agent reconnect.

Server Layer

Multiple yuzu-server replicas serve REST, HTMX, MCP, enrollment, command admission, and collation. Singleton jobs use Postgres advisory locks or row leases. Process-local state is cache only for HA-sensitive flows.

Postgres Layer

Each regional cell uses a single writable primary, synchronous same-region standby, async warm-region replica, PITR, tested backups, monitored replication lag, and an explicit manual promotion runbook. Multi-primary writes are out of scope for v1.

Correctness Invariants

Invariant Reason
Gateways own live streams only; Postgres owns accepted work. Gateway loss must not lose admitted commands.
Server replicas are interchangeable on HA paths. Dispatch and collate must survive process loss.
Presence and delivery are fenced by epoch and lease owner. Stale gateways cannot clobber newer ownership.
Delivery is at-least-once, with idempotent command and response recording. Exactly-once network delivery is not a realistic distributed-system contract.
Warm-region promotion fences the old primary before accepting writes. Prevents split-brain command, audit, and RBAC state.
Global views are aggregate-first and residency-aware. Prevents the global plane from becoming an undeclared data-exfiltration path.

Operational Requirements

  • SLOs for API/MCP availability, command admission, dispatch latency, gateway reconnect, mailbox age, command success, and DR readiness.
  • Dashboards for gateways, servers, Postgres, replication lag, command leases, reconnect rate, mailbox backlog, and error-budget burn.
  • Alerts for SLO burn, Postgres lag, gateway pressure, reconnect storms, lease backlog, stuck singleton jobs, failed backups, failed restore drills, and standby promotion readiness.
  • Runbooks for gateway loss, gateway drain, server loss, Postgres failover, warm-region promotion, reconnect storm, backup restore, regional isolation, and post-failover reconciliation.
  • Scale and chaos proof from 10k to 100k to 1.2M simulated agents.

Consequences

  • HA becomes a multi-quarter platform program, not a gateway-only feature.
  • HA-sensitive server state defaults to Postgres per ADR-0006/0007/0008.
  • Agent registration/config grows a signed gateway directory and regional assignment surface.
  • Gateway upstream protocol grows heartbeat, load, and drain publication.
  • Command dispatch moves from in-memory routing to durable mailbox ownership.
  • Operations artifacts become product deliverables: SLOs, runbooks, drills, dashboards, and evidence.

Options Considered

Option Decision
Single-region HA first Rejected. Insufficient for a large global enterprise and would force an immediate second architecture pass.
Global active primary Rejected. Creates latency, residency, and blast-radius problems for 56 countries.
Country-level cells from v1 Deferred. Macro-regional cells are the first target; strict country residency can extend the model.
Active-active multi-region writes Rejected for v1. Conflict semantics across commands, audit, RBAC, enrollment, and policy state are too costly.
Gateway gossip as durable truth Rejected. Gossip is useful for fast routing, but durable recovery needs leases and fencing in Postgres.

Implementation Ladder

  1. Define Enterprise HA SLOs, evidence requirements, and readiness semantics.
  2. Add signed gateway directory publication and agent-side multi-gateway failover.
  3. Add gateway heartbeat, load, and drain publication.
  4. Add durable gateway and agent presence with epoch fencing.
  5. Add durable command mailbox and per-target leases.
  6. Move bundle manifests to Postgres.
  7. Make server replicas horizontally safe by fencing singleton jobs.
  8. Package a 3-zone regional cell deployment.
  9. Add warm-region replication, promotion tooling, and standby readiness checks.
  10. Add the thin global aggregate/routing plane.
  11. Run scale and chaos proof against large-enterprise-size assumptions.
