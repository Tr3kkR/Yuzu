# Security review — SAML 2.0 Service Provider (SOC 2 CC6.1)

**Date:** 2026-07-01
**Change:** SAML 2.0 SP-initiated single sign-on — thin first slice against a
single, statically-pinned IdP. SP-initiated HTTP-Redirect AuthnRequest; ACS
validation via HTTP-POST; session seam mirrors the OIDC path.
**Branch:** `feat/auth-saml-sp`
**Controls:** SOC 2 **CC6.1** (logical access — enterprise SSO integration).
Closes `/auth-and-authz` gap-matrix **Workstream B P1 #6**.

## What shipped

- **`GET /auth/saml/start`** — builds a `<samlp:AuthnRequest>` (random `ID`,
  SP entity ID, ACS URL, `ForceAuthn=false`), sets the `__Host-yuzu_saml_bind`
  `Secure; HttpOnly; SameSite=None` binding cookie, and redirects the browser
  to the IdP via HTTP-Redirect binding (deflate-compressed, URL-encoded
  `SAMLRequest` query parameter).
- **`POST /saml/acs`** — validates the `<samlp:Response>` and mints a
  `yuzu_session` cookie on success.
- **Session:** `auth_source = "saml"`, `role = user` (all SAML users). In-memory
  and ephemeral, identical lifetime to OIDC sessions (8-hour absolute; optional
  `--session-inactivity-secs` idle timeout applies).
- **Audit:** `auth.saml_login` / `result=ok` on success;
  `auth.saml_login_failed` / `result=error` on any validation failure.
- **Configuration:** five flags (`--saml-idp-entity-id`, `--saml-idp-sso-url`,
  `--saml-idp-cert`, `--saml-sp-entity-id`, `--saml-sp-acs-url`). All five are
  required for correct operation and all five are validated at startup; a
  partial set leaves SAML disabled (fail-closed).
- **Platform gate:** SAML is Linux/macOS only. On Windows any SAML flag logs
  an error at startup and SAML is not enabled (fail-closed).
- **HTTPS gate:** if `--https-cert`/`--https-key` are not configured the server
  logs an error at startup and leaves SAML disabled (fail-closed).

## Security properties asserted

### P1 — Pinned-cert-only trust anchor; KeyInfo ignored
The IdP signing certificate is supplied via `--saml-idp-cert` and is the
**sole** trusted trust anchor. In-document `<ds:KeyInfo>` values are
intentionally ignored during signature verification (`xmlSecDSigCtx`). An
adversary cannot nominate a different key by embedding it in the SAML document.

### P2 — XSW (XML Signature Wrapping) two-layer binding
The validator locates the signed element by `ID` attribute reference (the
`<samlp:Response>` and, independently, the `<saml:Assertion>`) and verifies
that the element under the `<ds:Signature>` block matches the element that was
consumed. A wrapped unsigned sibling cannot be mistaken for a validated element.

### P3 — Replay protection via solicited-only, single-use InResponseTo
Each `AuthnRequest` carries a random `ID`; the ID is stored in process memory.
The ACS handler requires the `<samlp:Response>` `InResponseTo` attribute to
match an outstanding pending ID and consumes it on first use. A replayed
response is rejected (no matching pending ID). Unsolicited responses (no
`InResponseTo`) are also rejected — the solicited-only invariant is enforced.

### P4 — `__Host-` browser-binding cookie (forced-login / session-fixation)
The `__Host-yuzu_saml_bind` cookie (set by `GET /auth/saml/start`, cleared at
`POST /saml/acs` or on validation failure) binds the ACS POST back to the
browser that initiated the flow. The `__Host-` prefix mandates `Secure`,
`Path=/`, and no `Domain`, so a subdomain or plain-HTTP sibling cannot shadow
it. A POST to `/saml/acs` without the cookie is rejected with
`auth.saml_login_failed` / `result=error`.

### P5 — HTTPS fail-closed at startup
The binding cookie's `Secure` attribute causes browsers to silently drop it over
plain HTTP, making the CSRF guard inert. If `--https-cert`/`--https-key` are not
configured the server logs an error at startup and does not enable SAML.

### P6 — Windows fail-closed
On Windows builds the required XML processing library is not available in the
vcpkg manifest. Any SAML flag on a Windows server logs an error at startup;
`saml_provider_` is never constructed; all SAML routes return 404.

### P7 — Size caps (DoS guard)
Two independent caps protect against oversized input:
- **ACS POST body: 1 MiB** (`kSamlMaxBodyBytes = 1 048 576`) — checked before
  any parsing; oversized bodies are rejected with `auth.saml_login_failed` and
  a redirect to `/login?error=saml`.
- **IdP cert PEM: 64 KiB** (`kSamlCertMaxBytes = 65 536`) — checked at startup
  before the cert is loaded into the provider; an oversized file leaves SAML
  disabled (fail-closed).
- **Assertion XML: 1 MiB** — a second check inside `validate_response` after
  base64-decoding, before libxml2 parsing.

### P8 — XXE and DTD rejection
The XML parser is invoked with `XML_PARSE_NONET` (prevents external-entity
network fetches). After parsing, the validator checks for the presence of a
`<!DOCTYPE>` declaration in the parsed document and rejects any response that
carries one. No well-formed SAML response includes a DTD; its presence indicates
a crafted document (XXE attempt, entity injection, or fuzzer probe).

### P9 — Audience, recipient, and expiry validation
- `<saml:AudienceRestriction>` must include `--saml-sp-entity-id`.
- `<saml:SubjectConfirmationData Recipient>` must equal `--saml-sp-acs-url`.
- `NotOnOrAfter` on `<saml:Conditions>` and `<saml:SubjectConfirmationData>`
  are enforced against wall-clock time.

### P10 — RelayState open-redirect safety
The `RelayState` value is checked to be a same-origin relative path before it
is used as a redirect target: must begin with `/`, must not be protocol-relative
(`//`), must carry no backslash, no control characters, no non-ASCII bytes, and
no `..` path segment (evaluated after URL-decoding). All other values redirect
to the dashboard root.

## Findings addressed during review

- **Audit result tokens corrected.** Draft docs said `result=success`/`failure`;
  the code emits `result=ok`/`error` (matching the server's canonical
  audit-result vocabulary). Docs corrected.
- **False JIT-elevation claim removed.** Draft docs and the SAML session section
  claimed SAML users could gain admin via JIT elevation. Non-functional: the
  elevation endpoint checks `is_elevation_eligible` + `mfa_status` in `auth.db`;
  SAML users have no local `users` row — both lookups fail-closed. Corrected to
  state plainly that SAML users are permanently `role=user` in this release.
- **MFA step-up scope corrected.** Draft docs said "under `optional`/`admin-only`
  SAML sessions pass step-up gates." Not true: `require_mfa_step_up` returns 403
  for ALL SAML sessions unconditionally (before enforcement mode is consulted) —
  SAML users cannot access any of the 11 step-up-gated endpoints regardless of
  `--mfa-enforcement`. Docs corrected to reflect the actual gate behavior.
- **Login-button claim removed.** Draft docs described a "Sign in with SAML"
  button on the login page. No such button exists (`login_ui.cpp` unchanged).
  Corrected to: navigate to `GET /auth/saml/start`.
- **Startup validation count.** All five `--saml-*` flags, including
  `--saml-idp-entity-id`, are checked by the `saml_config_complete` predicate
  in `server.cpp`; a partial set of any of the five leaves SAML disabled at
  startup (fail-closed). Docs updated to describe five-flag startup validation.
- **HTTPS requirement made prominent.** The `Secure` cookie requirement was
  buried; a prominent callout was added to all three docs.
- **MFA enforcement incompatibility documented.** `--mfa-enforcement=required`
  is not supported with SAML (SAML users have no local MFA enrollment); docs
  corrected and a clear recommendation added.

## Residual risks (accepted)

- **In-process pending-ID state.** On a server restart, all outstanding
  `AuthnRequest` IDs are lost and any in-flight SAML flows will fail as
  "unsolicited". Fail-safe (the user re-authenticates). OIDC shares this
  behaviour.
- **AuthnRequest unsigned.** IdPs configured to require signed AuthnRequests will
  reject the SP's requests. Documented prerequisite; OIDC is the alternative.
- **SAML users cannot elevate.** JIT elevation requires a local `users` row;
  SAML users have none. Fail-closed (denied, not granted). Accepted as a v1
  limitation; group→role mapping is the tracked resolution.
