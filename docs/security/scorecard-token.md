# `SCORECARD_READ_TOKEN` — PAT for OpenSSF Scorecard

## What it is

A classic GitHub personal access token that `.github/workflows/scorecard.yml` passes to `ossf/scorecard-action` as `repo_token`. Without it, Scorecard can only read the public GitHub API surface and its `Branch-Protection` check caps at ~3/10 — it cannot see Repository Rulesets (which is how Yuzu actually protects `main`). With the PAT, `Branch-Protection` jumps to 8–9 on the next scheduled scan.

The secret is optional by design: the workflow falls back to `github.token` when it's absent, so fork runs and first-time-setup scans still succeed (with the reduced score).

## Required scopes

**Classic PAT — NOT fine-grained**, because Scorecard's ruleset-reading code paths still use the v3 REST endpoints that fine-grained tokens handle inconsistently.

| Scope | Why |
|---|---|
| `public_repo` | Read rulesets, branch protection, commit status, PR review history from the public repo |
| `read:org` | Let Scorecard detect organization-member contributors correctly (affects `Contributors` check) |

**Do NOT grant:** `repo` (includes private repos — over-broad), `write:*`, `admin:*`, `delete:*`, `user`, `gist`.

A token with any of those scopes is automatically rejected in review — the PAT must be generated with the exact two scopes above.

## Rotation

- **Cadence:** every 365 days, or immediately on any signal of compromise (unexpected workflow run, unauthorised access log, token leak).
- **Process:**
  1. Create a new PAT at https://github.com/settings/tokens (classic tokens section).
  2. Select the two scopes above. Set expiration to 90 days (GitHub hard-caps at 400).
  3. Update the `SCORECARD_READ_TOKEN` secret at https://github.com/Tr3kkR/Yuzu/settings/secrets/actions.
  4. Revoke the old PAT immediately — don't let it float.
  5. Trigger a manual Scorecard run via `gh workflow run scorecard.yml --ref main` and confirm the `analysis` job succeeds + the published score reflects the ruleset data.
- **Reminder:** add a repeating calendar event on the token creator's account 330 days ahead of expiry so rotation is never fire-drill.

## Who holds it

The token is generated by and stored under Nathan's account (repo owner). A future automation or shared org account that handles this is a better long-term home — track as follow-up work when the contributor base grows.

## Why a PAT and not a GitHub App

The Scorecard action's `repo_token` input expects a bearer token that the action passes to the GitHub API as `Authorization: token <value>`. GitHub App installation tokens are valid bearer values but require a signing-key-based exchange step per run, which `ossf/scorecard-action` does not orchestrate. If Sigstore / OSSF ships a GitHub-App-backed variant of the action in the future, migrate to it — less rotation overhead, installation-scoped permissions are finer-grained than PAT scopes.

## GitHub token format (stateless `ghs_`) — compatibility confirmed

GitHub is rolling out a new stateless, JWT-format installation/`GITHUB_TOKEN` value (`ghs_` prefix, up to **520 characters**) — see the [2026-05-15 changelog](https://github.blog/changelog/2026-05-15-github-app-installation-tokens-per-request-override-header/). **Audited 2026-06-22: Yuzu's entire GitHub token estate is unaffected.**

- **No GitHub App** anywhere in the repo — we never mint installation tokens (no `create-github-app-token`, no `POST /app/installations/:id/access_tokens`), so the temporary `X-GitHub-Stateless-S2S-Token` override header is irrelevant and there is nothing to test against it.
- **Every token is consumed opaquely.** `SCORECARD_READ_TOKEN`, the Actions `GITHUB_TOKEN`, `RUNNER_INVENTORY_TOKEN`, `GHCR_PAT`, `CLA_REPO_ACCESS_TOKEN`, and `gh auth token` are all handed straight to `gh`, `docker login --password-stdin`, or `ossf/scorecard-action` as bearer strings. No workflow or script parses, length-checks, regex-validates, truncates, or persists a GitHub token — so the longer format and `ghs_` prefix pass through transparently.
- **Product code is out of scope.** Yuzu's runtime never calls `api.github.com`. The `Authorization: Bearer` length caps in `auth.cpp` (64 hex session / 256 API token) govern Yuzu-issued tokens only, never GitHub's.

**The one place this could change:** if a future migration mints a GitHub App installation token here (per "Why a PAT and not a GitHub App" above), that new code path must treat the value as an opaque string up to 520 chars — never assume a fixed length or a `ghp_`/40-char shape.

## Pitfalls

1. **Expired PAT silently caps the score.** The workflow succeeds (falls back to `github.token`), but Branch-Protection drops back to ~3. Monitor the score weekly for 4 weeks after rotation — sudden drop = stale PAT.
2. **Fine-grained PAT doesn't work.** As of the 2026-04-20 Scorecard v5.x release, fine-grained PATs produce partial data — specifically ruleset reads fail with empty results but no error. Use a classic PAT only.
3. **Scope creep.** If a future Scorecard version requires additional scopes, the error message is opaque (`403 Resource not accessible by integration`). Check the `ossf/scorecard-action` release notes before widening scopes — most claims for wider scope come from misconfigured workflows, not actual scorecard requirements.
