# ADR-0001: Use msquic + quicer for QUIC transport

- **Status**: Accepted — pending spike validation (PR 0). If the spike
  fails any of the pass criteria below, status changes to Rejected and
  a new ADR with the alternative is required.
- **Accepted-by**: Nathan Dornbrook (project lead) on 2026-05-07
- **Date**: 2026-05-07
- **Tracking issue**: #376
- **Supersedes**: prior implicit decision to use gRPC (no ADR existed)
- **Related**: #375 (LNK2038/LNK2005 saga), #501 (absl DLL kSeed),
  `docs/per-target-agent-build-pipeline-scope.md`,
  `.claude/agents/build-ci.md` "Windows MSVC static-link history and #375"

## Context

Yuzu's RPC transport between agent ↔ server, server ↔ gateway, and gateway
↔ agent has been gRPC since project inception. The wire format is
protobuf; the framing and flow control come from gRPC over HTTP/2.

The C++ side of this brings in a heavy dependency closure via vcpkg —
`grpc`, `abseil`, `c-ares`, `re2`, `upb`, `utf8-range` — which has been a
recurring source of build outages:

1. **#375** (2026-04-10 → 2026-04-14): LNK2038 / LNK2005 saga on Windows
   MSVC. Three failed fix attempts, ~4 days of CI red. Resolved via the
   "option D" hand-wire block in `meson.build` (lines ~91–240) plus a
   static-linkage override in `triplets/x64-windows.cmake`.
2. **#501**: Abseil's `__hash.kSeed` becomes a different value in each
   DLL when abseil is linked dynamically, breaking `absl::flat_hash_*`
   across DLL boundaries. Multiple test workarounds.
3. **2026-04-28 vcpkg cache nuke**: Exposed CLI11 + Catch2 as the next two
   dominoes after the option-D pattern was set; they were hand-wired the
   same way.
4. **Alpine/musl ambition**: `docs/per-target-agent-build-pipeline-scope.md`
   (2026-05-07) determined that abseil, grpc, re2, c-ares, upb collectively
   have no usable musl story in vcpkg, blocking Alpine support for the
   agent. Phase 2 of that plan is gated on this migration.

The Erlang gateway side has parallel issues: `grpcbox` pulls in
`chatterbox` (HTTP/2 implementation that has been a recurring source of
brittleness) plus `gproc`. The grpcbox sub-tree is non-trivial.

The strategic option captured in #376 is to replace gRPC with **QUIC**,
retaining protobuf as the wire format. QUIC gives us reliable multiplexed
bidirectional streams (the same shape gRPC uses) without the C++ ABI /
CMake ecosystem tax of gRPC.

## Decision

We will use **msquic** on the C++ side and **quicer** (Emqx's NIF wrapper
around msquic) on the Erlang gateway side.

Both ends terminate the same QUIC implementation. No QUIC-frame-level
heterogeneity between agent/server (C++) and gateway (Erlang).

Wire format remains protobuf, generated by the existing `proto/gen_proto.py`
toolchain. Service definitions in `proto/yuzu/{agent,server,gateway}/v1/`
are unchanged.

Each logical RPC is mapped to a single QUIC bidirectional stream. Each
direction frames protobuf messages with a 4-byte big-endian length prefix.
Trailing gRPC-style status/metadata becomes a final protobuf message
(`yuzu.transport.v1.TrailingStatus`) on the stream before half-close.

mTLS is preserved via msquic's TLS API.

## Alternatives considered

### lsquic (LiteSpeed)

- **Pros**: Mature C library, used in production by LiteSpeed Web Server.
  Includes HTTP/3 layer (not relevant for us but indicates depth).
- **Cons**: No first-class Erlang binding. Would require us to maintain
  custom NIF on the gateway side or commission/port an Erlang wrapper.
  Heterogeneous stack (lsquic on C++ side, something else on Erlang) is
  exactly the QUIC-frame-level interop risk we want to avoid.
- **Rejected** because of the Erlang-side gap.

### ngtcp2 + nghttp3

- **Pros**: Reference-quality C QUIC implementation; widely used by
  curl-side ecosystems. Cleanly separable QUIC / HTTP/3 layers.
- **Cons**: Same Erlang-side gap as lsquic — no maintained NIF binding.
  Would need to be paired with quicer or a custom binding regardless,
  which means two QUIC implementations in production. Library API is
  lower-level than msquic's; more boilerplate to wire bidi streams +
  flow control.
- **Rejected** because the Erlang-side gap forces a heterogeneous stack
  that we explicitly want to avoid.

### picoquic

- **Pros**: Research-grade QUIC implementation with strong reputation
  for correctness. Used in IETF interop testing.
- **Cons**: Marketed as a research/experimentation library, not a
  production-deployment library. No first-class Erlang binding.
  Maintenance cadence uncertain.
- **Rejected** as not production-grade.

### quiche (Cloudflare)

- **Pros**: Production-grade Rust QUIC implementation with C ABI.
  Used by Cloudflare's edge.
- **Cons**: Rust dep adds toolchain complexity (we are otherwise C++23).
  Erlang binding situation comparable to lsquic — none maintained at
  first-class. Cloudflare's primary use case (HTTP/3 proxy front-end)
  is not aligned with our use case (long-lived control-plane bidi
  streams).
- **Rejected** because of the Rust toolchain addition and Erlang gap.

### Stay on gRPC

- **Pros**: Already working. No migration risk. No customer-facing impact.
- **Cons**: Permanent vcpkg+abseil+cmake-method tax. Alpine/musl support
  remains blocked. Future vcpkg upgrades continue to surprise. The
  cumulative cost of the recurring fire-fights (#375, #501,
  2026-04-28 cache nuke) outweighs the migration cost on a multi-year
  view.
- **Rejected** because the recurring tax is unsustainable and Alpine
  support is on the roadmap.

## Why msquic specifically

- Microsoft-maintained, used by Windows Server and .NET — production
  grade, not research.
- vcpkg port exists. Conan recipe exists. msquic is C, doesn't pull
  abseil — none of the gRPC-class dep-graph problems.
- Cross-platform: Windows / Linux / macOS via the same C ABI. Linux
  build uses OpenSSL by default; Windows uses schannel by default; we
  pin OpenSSL on Windows too for symmetry with the cert-store contract.
- Has an actively maintained Erlang binding (quicer) by Emqx — no
  orphan-binding risk.

### Cryptographic module posture (Workstream C)

The current gRPC stack uses BoringSSL (vcpkg's gRPC port links its
embedded copy). Post-migration the C++ side uses msquic with OpenSSL
3.x on Linux/macOS and on Windows (we pin away from schannel for
symmetry). The Erlang gateway side already uses OpenSSL via OTP's
crypto application.

Yuzu does not currently claim FIPS 140-3 validation. If a future
customer requires FIPS-validated TLS, OpenSSL 3.x ships a FIPS provider
that satisfies that bar with explicit configuration; the migration
preserves the option without committing to it. BoringSSL has its own
FIPS-validated build (BoringCrypto) used by some downstream consumers,
but Yuzu has never relied on it.

This posture is a customer-questionnaire-relevant change. The follow-up
issue tracking the questionnaire-update package (CAIQ encryption row,
architecture whitepaper transport section, DPA "encryption in transit"
annex) is referenced in the consequences section below.

## Why quicer specifically

- Maintained by Emqx, a high-load production user (MQTT broker over QUIC).
- NIF wraps msquic directly, so the Erlang gateway and the C++ stack
  speak the same QUIC implementation byte-for-byte at the frame level.
- Active development cadence. Used in production at scale.

The primary risk with a NIF-based binding is that a bug in the NIF can
crash the BEAM VM. We mitigate this by isolating quicer-using processes
under their own simple_one_for_one supervisor and ensuring dialyzer is
clean across the whole gateway after the migration.

## Consequences

### Positive

- Eliminates `grpc`, `abseil`, `c-ares`, `re2`, `upb`, `utf8-range` from
  the C++ dep graph (~6 deps).
- Eliminates `grpcbox`, `chatterbox`, `gproc` from the Erlang dep graph
  (3 deps).
- Removes the `meson.build` Windows option-D hand-wire block (~150 lines).
- Removes the `triplets/x64-windows.cmake` static-linkage override.
- Removes the #501 absl DLL kSeed cross-DLL workarounds (becomes obsolete
  once abseil is gone).
- Unblocks Alpine/musl support — `docs/per-target-agent-build-pipeline-
  scope.md` Phase 2.
- Reduces binary size; expected ≥ 30 % reduction post-migration.
- Reduces Windows MSVC debug link time; expected ≥ 50 % reduction post-
  migration (option-D hand-wire was a major contributor).

### Negative / costs

- ~6–9 weeks of dedicated engineering effort across 11 PRs
  (`docs/per-target-agent-build-pipeline-scope.md` references the same
  effort estimate from #376).
- Greenfield C++ on QUIC primitives — risk of subtle bugs at first.
- Greenfield Erlang on quicer — risk of NIF crashes in early use.
- Dual-stack soak period (PRs 5–7) ⇒ for ~3 release cycles we maintain
  both transports. Operational complexity during that window.
- Long-lived `feat/quic-transport` branch off `dev` — must be
  rebased/merged from `dev` weekly to avoid drift.

### Mitigation matrix

| Risk | Mitigation |
|---|---|
| msquic ↔ quicer QUIC-frame interop divergence | PR 0 spike validates before any production code; pin both sides to compatible msquic versions |
| Bidi-stream semantics: gRPC half-close + status trailers vs raw QUIC streams | Encode trailing-status as final protobuf message (`TrailingStatus`); document explicitly in `transport.hpp` |
| mTLS cert rotation on msquic vs grpc::SslCredentials | PR 3 explicitly tests `cert_reloader` rotation; design `Credentials` interface to expose a reload callback; cert lifecycle policy doc required pre-PR 3 merge (Workstream B) |
| quicer NIF crash takes BEAM VM down | Isolate quicer-using processes under their own supervisor; dialyzer-clean is mandatory before merge |
| Agent fleet rollout breaks production | PR 5 dual-stack ⇒ PR 6 default-flip ⇒ rollback is just an agent flag change with server still dual-listening |
| Customer enterprise networks block UDP/443 (firewalls, Zscaler, Netskope) | Documented `--transport grpc` flag preserves the old transport; `yuzu-agent --probe-transport` subcommand surfaces UDP reachability before fleet-wide flip; required pre-PR 5 merge |
| DPI / TLS-interception middleboxes (Bluecoat, Symantec ProxySG) cannot inspect QUIC handshake | Document operator bypass-list procedure for the Yuzu control-plane FQDN; add to security-hardening doc pre-PR 5; pilot-customer hand-test gate validates each design partner's DPI environment before default-flip |
| Compliance questionnaire answer becomes stale (BoringSSL → OpenSSL/msquic) | Follow-up issue tracks questionnaire-update package (CAIQ encryption row, architecture whitepaper transport section, DPA "encryption in transit" annex, shared-responsibility matrix network-layer row) due no later than PR 6 (default-flip) merge |

## Spike validation gate (PR 0)

Before any production code is written, PR 0 includes a throwaway spike
that stands up:

- A C++ msquic server listening on `localhost:50053`
- An Erlang quicer client connecting to it
- A bidirectional echo: each end writes 1 KiB protobuf-encoded messages
  in both directions for 30 seconds
- Server and client both terminate cleanly on half-close

Pass criteria:
- TLS 1.3 handshake succeeds with self-signed cert pair
- Bidi stream survives 30 s of two-way writes with zero loss
- Half-close from either end is detected by the other within 1 s
- Flow control: writer that blocks for 5 s on a slow reader resumes
  cleanly when reader catches up

Fail mode: any of the above. If the spike fails, halt the migration
plan, re-evaluate library choice (probable next move: lsquic + custom
Erlang NIF, or stay on gRPC and re-scope), and update this ADR with
status `Rejected` and a new ADR with the alternative.

### Evidence capture (Workstream D)

The spike output (msquic server log, quicer client log, frame-counter
totals, half-close timing, slow-reader resume timing) is redirected to
files under `docs/spike-results/0001-quic-transport/` (or attached as
artifacts to issue #376) and committed to the repo. The retention
target is the lifetime of the migration — these files are
audit-evidence for the SOC 2 change-management trail (Workstream F)
proving the architectural decision was validated before commitment.

A separate `docs/spike-results/0001-quic-transport/README.md` summarises
which environment ran the spike (host OS, msquic version, quicer
version, cert-pair fingerprint), who ran it, and pass/fail per
criterion. The named approver in this ADR's header signs off on the
evidence by referencing the spike-results commit SHA in their approval.

## Release communication checklist

Each implementation PR in the ladder carries explicit
operator-communication obligations:

- **PR 5 (dual-stack feature flag)**:
  - CHANGELOG `[Unreleased] – Added` entry naming the new
    `--transport=grpc|quic|auto` flag
  - `docs/user-manual/security-hardening.md` Network Requirements
    section adding UDP/50051 (or whichever ports apply)
  - `docs/user-manual/upgrading.md` version-table row documenting that
    server $X+1 dual-listens, agents may stay on $X
  - `--probe-transport` subcommand documented in the same section
  - **Pilot-customer hand-test gate**: 1–3 design-partner customers
    validate dual-stack in their actual enterprise networks (including
    DPI/proxy infrastructure) for ≥ 1 week before PR 6 may merge

- **PR 6 (agent default-flip to QUIC)**:
  - CHANGELOG `Breaking` entry: agent default transport changes
  - upgrading.md version-table row noting rollback path
  - Compliance questionnaire-update issue closed (CAIQ + whitepaper +
    DPA + SRM)

- **PR 7 (gateway default-flip)**:
  - CHANGELOG `Breaking` entry
  - upgrading.md version-table row

- **PR 8 (gRPC removal)**:
  - CHANGELOG `Breaking` entry: agents older than this version cannot
    connect
  - CLAUDE.md target-architecture diagram updated (gRPC label → QUIC)
  - `docs/per-target-agent-build-pipeline-scope.md` Status updated to
    "Phase 2 unblocked"

The governance pipeline runs on each PR; the docs-writer agent
verifies the relevant items above land in the same commit as the
behaviour change.
