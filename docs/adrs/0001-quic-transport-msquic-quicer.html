ADR-0001: Use msquic + quicer for QUIC transport

  • Status: Accepted — spike validation passed 2026-05-07 (docs/spike-results/0001-quic-transport/README.md).
  • Accepted-by: Nathan Dornbrook (project lead) on 2026-05-07. Approval references the spike-results files committed in af706ea and the surrounding governance hardening rolled into governance round 5 (this branch); both are in-tree under git tamper-evidence.
  • Date: 2026-05-07
  • Tracking issue: #376
  • Supersedes: prior implicit decision to use gRPC (no ADR existed)
  • Related: #375 (LNK2038/LNK2005 saga), #501 (absl DLL kSeed), docs/per-target-agent-build-pipeline-scope.md, .claude/agents/build-ci.md "Windows MSVC static-link history and #375", docs/spike-results/0001-quic-transport/ (PR 0 evidence)

Context

Yuzu's RPC transport between agent ↔ server, server ↔ gateway, and gateway ↔ agent has been gRPC since project inception. The wire format is protobuf; the framing and flow control come from gRPC over HTTP/2.

The C++ side of this brings in a heavy dependency closure via vcpkg — grpc, abseil, c-ares, re2, upb, utf8-range — which has been a recurring source of build outages:

  1. #375 (2026-04-10 → 2026-04-14): LNK2038 / LNK2005 saga on Windows MSVC. Three failed fix attempts, ~4 days of CI red. Resolved via the "option D" hand-wire block in meson.build (lines ~91–240) plus a static-linkage override in triplets/x64-windows.cmake.
  2. #501: Abseil's __hash.kSeed becomes a different value in each DLL when abseil is linked dynamically, breaking absl::flat_hash_* across DLL boundaries. Multiple test workarounds.
  3. 2026-04-28 vcpkg cache nuke: Exposed CLI11 + Catch2 as the next two dominoes after the option-D pattern was set; they were hand-wired the same way.
  4. Alpine/musl ambition: docs/per-target-agent-build-pipeline-scope.md (2026-05-07) determined that abseil, grpc, re2, c-ares, upb collectively have no usable musl story in vcpkg, blocking Alpine support for the agent. Phase 2 of that plan is gated on this migration.

The Erlang gateway side has parallel issues: grpcbox pulls in chatterbox (HTTP/2 implementation that has been a recurring source of brittleness) plus gproc. The grpcbox sub-tree is non-trivial.

The strategic option captured in #376 is to replace gRPC with QUIC, retaining protobuf as the wire format. QUIC gives us reliable multiplexed bidirectional streams (the same shape gRPC uses) without the C++ ABI / CMake ecosystem tax of gRPC.

Decision

We will use msquic on the C++ side and quicer (Emqx's NIF wrapper around msquic) on the Erlang gateway side.

Both ends terminate the same QUIC implementation. No QUIC-frame-level heterogeneity between agent/server (C++) and gateway (Erlang).

Wire format remains protobuf, generated by the existing proto/gen_proto.py toolchain. Service definitions in proto/yuzu/{agent,server,gateway}/v1/ are unchanged.

Each logical RPC is mapped to a single QUIC bidirectional stream. Each direction frames protobuf messages with a 4-byte big-endian length prefix. Trailing gRPC-style status/metadata becomes a final protobuf message (yuzu.transport.v1.TrailingStatus) on the stream before half-close.

mTLS is preserved via msquic's TLS API.

Alternatives considered

lsquic (LiteSpeed)

  • Pros: Mature C library, used in production by LiteSpeed Web Server. Includes HTTP/3 layer (not relevant for us but indicates depth).
  • Cons: No first-class Erlang binding. Would require us to maintain custom NIF on the gateway side or commission/port an Erlang wrapper. Heterogeneous stack (lsquic on C++ side, something else on Erlang) is exactly the QUIC-frame-level interop risk we want to avoid.
  • Rejected because of the Erlang-side gap.

ngtcp2 + nghttp3

  • Pros: Reference-quality C QUIC implementation; widely used by curl-side ecosystems. Cleanly separable QUIC / HTTP/3 layers.
  • Cons: Same Erlang-side gap as lsquic — no maintained NIF binding. Would need to be paired with quicer or a custom binding regardless, which means two QUIC implementations in production. Library API is lower-level than msquic's; more boilerplate to wire bidi streams + flow control.
  • Rejected because the Erlang-side gap forces a heterogeneous stack that we explicitly want to avoid.

picoquic

  • Pros: Research-grade QUIC implementation with strong reputation for correctness. Used in IETF interop testing.
  • Cons: Marketed as a research/experimentation library, not a production-deployment library. No first-class Erlang binding. Maintenance cadence uncertain.
  • Rejected as not production-grade.

quiche (Cloudflare)

  • Pros: Production-grade Rust QUIC implementation with C ABI. Used by Cloudflare's edge.
  • Cons: Rust dep adds toolchain complexity (we are otherwise C++23). Erlang binding situation comparable to lsquic — none maintained at first-class. Cloudflare's primary use case (HTTP/3 proxy front-end) is not aligned with our use case (long-lived control-plane bidi streams).
  • Rejected because of the Rust toolchain addition and Erlang gap.

Stay on gRPC

  • Pros: Already working. No migration risk. No customer-facing impact.
  • Cons: Permanent vcpkg+abseil+cmake-method tax. Alpine/musl support remains blocked. Future vcpkg upgrades continue to surprise. The cumulative cost of the recurring fire-fights (#375, #501, 2026-04-28 cache nuke) outweighs the migration cost on a multi-year view.
  • Rejected because the recurring tax is unsustainable and Alpine support is on the roadmap.

Why msquic specifically

  • Microsoft-maintained, used by Windows Server and .NET — production grade, not research.
  • vcpkg port exists. Conan recipe exists. msquic is C, doesn't pull abseil — none of the gRPC-class dep-graph problems.
  • Cross-platform: Windows / Linux / macOS via the same C ABI. Linux build uses OpenSSL by default; Windows uses schannel by default; we pin OpenSSL on Windows too for symmetry with the cert-store contract.
  • Has an actively maintained Erlang binding (quicer) by Emqx — no orphan-binding risk.

Cryptographic module posture (Workstream C)

The current gRPC stack uses BoringSSL (vcpkg's gRPC port links its embedded copy). Post-migration the C++ side uses msquic with quictls/openssl 3.1.7-quic1 on Linux/macOS and on Windows (we pin away from schannel for symmetry). The Erlang gateway side continues to use upstream OpenSSL via OTP's crypto application.

The fork distinction matters for compliance: - quictls/openssl 3.1.7-quic1 is a fork of upstream OpenSSL 3.1.x maintained by the QUIC ecosystem to add the QUIC API extensions removed from the upstream 3.2 line. vcpkg's msquic 2.4.8 port vendors it directly. It is NOT the upstream OpenSSL 3.x line. - The fork does not share an upstream-coordinated CVE disclosure timeline with OpenSSL. A vulnerability patched in upstream 3.1.x may be patched in the fork at a later date; Yuzu's exposure window is bounded by the fork's release cadence and our vcpkg msquic port-bump latency. CVE monitoring posture for the fork is part of the questionnaire-update package (#891). - Customer-questionnaire answers MUST cite the fork explicitly, not "OpenSSL 3.x".

Yuzu does not currently claim FIPS 140-3 validation. If a future customer requires FIPS-validated TLS, upstream OpenSSL 3.x ships a FIPS provider that satisfies that bar with explicit configuration; the quictls fork is NOT FIPS-validated. The migration preserves the upstream-OpenSSL FIPS option for a future re-platform without committing to it. BoringSSL has its own FIPS-validated build (BoringCrypto) used by some downstream consumers, but Yuzu has never relied on it.

This posture is a customer-questionnaire-relevant change. Tracking issue: #891 — covers CAIQ encryption row, architecture whitepaper transport section, DPA "encryption in transit" annex, SRM network-layer row, and the fork-CVE-monitoring posture. Hard gate before PR 6 merge.

Why quicer specifically

  • Maintained by Emqx, a high-load production user (MQTT broker over QUIC).
  • NIF wraps msquic directly, so the Erlang gateway and the C++ stack speak the same QUIC implementation byte-for-byte at the frame level.
  • Active development cadence. Used in production at scale.

The primary risk with a NIF-based binding is that a bug in the NIF can crash the BEAM VM. We mitigate this by isolating quicer-using processes under their own simple_one_for_one supervisor and ensuring dialyzer is clean across the whole gateway after the migration.

Consequences

Positive

  • Eliminates grpc, abseil, c-ares, re2, upb, utf8-range from the C++ dep graph (~6 deps).
  • Eliminates grpcbox, chatterbox, gproc from the Erlang dep graph (3 deps).
  • Removes the meson.build Windows option-D hand-wire block (~150 lines).
  • Removes the triplets/x64-windows.cmake static-linkage override.
  • Removes the #501 absl DLL kSeed cross-DLL workarounds (becomes obsolete once abseil is gone).
  • Unblocks Alpine/musl support — docs/per-target-agent-build-pipeline- scope.md Phase 2.
  • Reduces binary size; expected ≥ 30 % reduction post-migration.
  • Reduces Windows MSVC debug link time; expected ≥ 50 % reduction post- migration (option-D hand-wire was a major contributor).

Negative / costs

  • ~6–9 weeks of dedicated engineering effort across 11 PRs (docs/per-target-agent-build-pipeline-scope.md references the same effort estimate from #376).
  • Greenfield C++ on QUIC primitives — risk of subtle bugs at first.
  • Greenfield Erlang on quicer — risk of NIF crashes in early use.
  • Dual-stack soak period (PRs 5–7) ⇒ for ~3 release cycles we maintain both transports. Operational complexity during that window.
  • Long-lived feat/quic-transport branch off dev — must be rebased/merged from dev weekly to avoid drift.

Per-call dispatcher-thread fanout at enterprise scale (governance UP-14) — PR 1b-3's bidi server dispatcher spawns one std::thread per active bidi RPC. At 100K Subscribe streams (one per agent), Linux default 8 MiB stack means ~800 GiB virtual; even a 256 KiB tuned stack yields ~25 GiB and a fragile thread-creation ceiling. Option (a) shipped: GrpcServerListener now owns a fixed-size pool sized at start() from ListenerOptions::bidi_dispatcher_pool_size (default clamp(64, hardware_concurrency × 8, 4096)). Saturation rejects with StatusCode::ResourceExhausted "transport: bidi dispatcher saturated" via the existing UnaryPendingFinish reaper path (preserves UP-1 lifetime invariant). on_bidi_finish no longer joins per-call threads — closes UP-13 head-of-line block on cq_worker. Operator guidance in docs/user-manual/server-admin.md direct-connect-vs-gateway sizing section. systemd TasksMax=infinity + LimitNPROC=infinity and Docker compose nproc: 65536 / pids_limit: -1 provide cgroup-level headroom. Tests at tests/unit/transport/test_transport_smoke.cpp [transport][bidi][pool]: pool reuse across more sequential calls than pool_size, overflow → ResourceExhausted, default size absorbs moderate concurrency. Gateway-mode deployment further reduces server-side pressure since the gateway (Erlang processes, lightweight) terminates Subscribe and only per-gateway concurrency reaches the C++ pool.
DownloadUpdate idle-read pins a bidi dispatcher pool slot indefinitely (governance UP-8 / #902) — pre-#902 the lifted DownloadUpdate handler at agent_service_impl.cpp blocks on read_pb for the request frame with no timeout. A misbehaving or stalled client that opens the stream and never writes would hold a pool slot until peer half-close or external cancel. Combined with #904's bounded pool, idle clients monopolise the pool and operators see ResourceExhausted rejections on legitimate concurrent downloads. Mitigated by BidiStream::read idle-deadline (#902, 2026-05-09). The pure virtual gained an optional std::chrono::milliseconds deadline parameter (default zero = no deadline, mirroring CallContext::deadline). On expiry, the implementation cancels the stream and surfaces StatusCode::DeadlineExceeded via final_status() on the side that armed the deadline. DownloadUpdate now arms the request-frame read with a 30 s deadline. Server-side handler distinguishes deadline expiry from peer-close by querying stream.final_status() after read() returns false. The matching read_pb(stream, out, deadline) template overload in proto_adapter.hpp keeps the adapter surface symmetric. Wire semantics caveat documented in the BidiStream contract block: the gRPC backend's TryCancel path means the peer observes Cancelled regardless of handler-supplied final status; the local DeadlineExceeded signal is what gates the slot release, not the wire-trailer reason. Tests at [transport][bidi][deadline]: server-side timeout, client-side timeout, happy-path no-fire.
Slow-write peer pins a bidi dispatcher pool slot during chunk streaming (governance UP-101 / #911) — UP-8's read deadline only bounds the FIRST read on a bidi stream. The chunk-streaming phase of DownloadUpdate (server writes 64 KiB chunks in a tight loop) re-stalls the dispatcher slot when a peer's TCP receive window collapses, NAT/firewall holds the stream, or HTTP/2 flow control window collapses. Combined with #904's bounded pool, slow-write peers re-introduce the UP-8 saturation surface at the streaming phase. Mitigated by BidiStream::write per-frame deadline (#911, 2026-05-10). The pure virtual gained an optional std::chrono::milliseconds deadline parameter symmetric with the read side. Default zero preserves unbounded; negative values clamp to zero (#915). Mechanism is identical to read: cv_.wait_for(lock, deadline, pred) + on-expiry set-flag + TryCancel + return-false. final_status() promotes to DeadlineExceeded on either flag. Server-side DownloadUpdate arms each chunk write with the operator-tunable ota_chunk_write_deadline_secs_ (default 30 s); agent-side Updater arms request-frame write with kRequestWriteDeadline = 5 s. Counter yuzu_grpc_requests_total{status="chunk_write_deadline_exceeded"} + agent-side yuzu_agent_ota_chunk_deadline_total{phase} (#924 / SRE-2) close CC7.2 evidence asymmetry. Tests at [transport][bidi][deadline][write]. Slow-link compound lockout (UP-206) closed by #934 (next row).
Slow-link compound lockout — chunk-write deadline + per-peer rate limit (governance UP-206 / #934) — A genuinely slow legitimate peer (satellite, residential 4G/LTE, congested corp WAN) hits the 30 s chunk-write deadline (#911) mid-OTA, retries on the next heartbeat, and burns through the per-peer rate-limit bucket (#913) within 5 cycles. Bucket refills at 1 token / 90 min, so the agent is locked out of OTA for 7.5 h despite being mostly-healthy. Mitigated by operator-tunable deadline + chunk-write-deadline token refund (#934, 2026-05-10). Two-part fix: (1) --ota-chunk-write-deadline-secs <n> / YUZU_OTA_CHUNK_WRITE_DEADLINE_SECS env (default 30, <= 0 clamps to default) lets operators raise the deadline to fit slow links — threaded through Config::ota_chunk_write_deadline_seconds → AgentServiceImpl::set_ota_chunk_write_deadline() → atomic member read once per call (snapshotted so reload doesn't reshape in-flight). (2) AgentServiceImpl::refund_download_update(peer) credits the consumed token back when the chunk-write deadline fires; refund caps at bucket capacity, empty-peer/unseen-peer no-op. New metric label yuzu_grpc_requests_total{status="rate_limit_token_refunded"} tracks the refund path so dashboards can split chunk-write-deadline failures into "charged" vs "refunded" — refund-counter MUST track chunk-write-deadline-counter exactly; divergence signals a regression. Threat model unchanged: fast-valid monopolisers still drain the bucket (their calls succeed, no chunk-write-deadline → no refund). Tests at [agent_service][rate_limit][refund].
Valid-mTLS insider monopolises bidi pool with rapid DownloadUpdate calls (governance UP-116 / #913) — #904's bounded pool + #902 / #911 deadlines cap slot-seconds, but a fast-valid client (mTLS-authed insider) can saturate the pool with legitimate-looking calls. Per-peer counter is needed to limit concurrency by identity. Mitigated by per-peer DownloadUpdate token bucket (#913, 2026-05-10). Token bucket in AgentServiceImpl::admit_download_update keyed by verified SAN identity (falls back to peer_uri in dev/UAT). Capacity 5 tokens, refill ~1 token / 90 minutes (= update_check_interval / 4). Excess requests return ResourceExhausted "per-peer DownloadUpdate rate limit exceeded". Counter yuzu_grpc_requests_total{status="rate_limited_per_peer"} + spdlog::warn with peer attribution. State is process-local (resets on restart) — fleet-monopolisation guard, not persistent quota. Multi-server HA bucket-sharing gap tracked at #935; bucket-map cap and size-metric also #935. Tests at [agent_service][rate_limit].

Three Gate 6 reviewers (compliance + sre + enterprise-readiness) converged on this as BLOCKING for pre-pilot tagging in PR 1c-4 governance. Spike validation gate (PR 0)

Before any production code is written, PR 0 includes a throwaway spike that stands up:

  • A C++ msquic server listening on localhost:50053
  • An Erlang quicer client connecting to it
  • A bidirectional echo: each end writes 1 KiB protobuf-encoded messages in both directions for 30 seconds
  • Server and client both terminate cleanly on half-close

Pass criteria: - TLS 1.3 handshake succeeds with self-signed cert pair - Bidi stream survives 30 s of two-way writes with zero loss - Half-close from either end is detected by the other within 1 s - Flow control: writer that blocks for 5 s on a slow reader resumes cleanly when reader catches up

Fail mode: any of the above. If the spike fails, halt the migration plan, re-evaluate library choice (probable next move: lsquic + custom Erlang NIF, or stay on gRPC and re-scope), and update this ADR with status Rejected and a new ADR with the alternative.

Evidence — 2026-05-07 run

Macro-summary (full detail in docs/spike-results/0001-quic-transport/README.md):

Pass criterion Observed
TLS 1.3 handshake 4 ms; ALPN yuzu-spike negotiated
Bidi 30 s, zero loss 15 000 chunks each direction, loss_bytes = 0
Half-close < 1 s client peer-FIN observed in 0 ms (sub-millisecond)
Flow control resume slow_pause_observed_ms = 5005, zero loss across the pause

Environment: macOS 26.4 / Apple Silicon arm64, AppleClang 21.0, OTP 28 / erts 16.4, msquic 2.4.8 (vcpkg port arm64-osx), quicer 0.2.15 (hex). Spike repo SHA at run time: 054cda0.

Two design choices that the production transport (transport/) must honour, pulled forward from spike findings (Caveats section of the spike-results README; ordering kept identical between this section and the README so a reader cross-referencing them does not see drift):

  1. quicer:async_send/3 without ?QUICER_SEND_FLAG_SYNC does not deliver send_complete events to the controlling process (gated at quicer/c_src/quicer_stream.c:1174 if (send_ctx->is_sync)). Application-level back-pressure tracking must use send/2 (sync) or async_send with the SYNC bit.
  2. quicer 0.2.x event tuples are 4-element {quic, Tag, Resource, Prop} even when the event has no payload (the shape is built at quicer/c_src/quicer_nif.c:1710 via enif_make_tuple4). 3-tuple patterns silently fall through to catchalls.

Both constraints are tracked for PR 4 (Erlang gateway QUIC migration) at #895 so the production gateway author has the complete patterns-to- inherit + patterns-to-NOT-inherit list before writing the production quicer client.
