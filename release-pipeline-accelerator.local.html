Release Pipeline Accelerator

Problem Statement

The v0.8.0-rc0 release took ~15 hours across two days. Every failure was discovered sequentially: push tag, wait 30+ min, read logs, fix, re-tag. The goal is bug fix to release in minutes, not days.

What Mature Projects Do

Dependency Management

  • Rust/Go: Language toolchains cross-compile natively; no vcpkg-style dependency builds
  • Kubernetes: Uses goreleaser for cross-compilation in one step
  • Grafana: Pre-built Docker images with all dependencies; runner is just a Docker host

Build Acceleration

  • sccache (Mozilla): Drop-in ccache replacement, supports MSVC/GCC/Clang, S3/local disk backend, shares cache between CI and local dev
  • vcpkg binary caching: VCPKG_BINARY_SOURCES="clear;files,/path,readwrite" — content-addressed cache by triplet+port hash
  • ccache: Already in use, but cold on reconfigured runner

Local Validation

  • Kubernetes: make verify mirrors CI checks locally
  • Grafana: make release-test simulates release pipeline
  • Common pattern: CI workflow calls a script; developers call the same script locally

Runner Management

  • Nix flakes: nix develop gives identical environment everywhere
  • Ansible: Playbook declares packages, paths, permissions
  • Docker-based runners: Container built from Dockerfile with everything pre-installed

Recommendations (Priority Order)

P0: .gitattributes (15 min, prevents CRLF class entirely)

* text=auto eol=lf
*.sh text eol=lf
*.py text eol=lf
*.erl text eol=lf
*.hrl text eol=lf
*.bat text eol=crlf
*.ps1 text eol=crlf
*.sln text eol=crlf
*.vcxproj text eol=crlf
*.exe binary
*.dll binary
*.so binary
*.dylib binary

P1: Runner Provisioning Script (2-3 hrs)

Create scripts/provision-runner.sh that: 1. Verifies all required tools are installed and in PATH 2. Verifies VCPKG_ROOT is set and writable 3. Verifies vcpkg binary cache is warm 4. Verifies group ownership (docker group) on shared dirs 5. Optionally installs missing tools (with sudo)

Create companion Ansible playbook infra/runner-provision.yml for full provisioning.

Run the verify-only mode as the first step of every CI workflow.

P2: vcpkg Binary Cache on Self-Hosted Runners (30 min)

Add to runner .env: 

VCPKG_BINARY_SOURCES=clear;files,/var/cache/vcpkg,readwrite

Create the cache dir with docker group ownership. The self-hosted runner keeps the cache permanently (no GHA cache eviction). After first build, vcpkg install becomes a ~10s cache restore.

P3: VCPKG_BUILD_TYPE=release on All Triplets (done)

Already applied to x64-linux-static.cmake and x64-windows.cmake. Halves vcpkg build time by skipping debug variants.

P4: Release Preflight Script (2-3 hrs)

Create scripts/release-preflight.sh: 

#!/usr/bin/env bash
set -euo pipefail
TAG="${1:?Usage: release-preflight.sh v0.8.0-rc0}"
VERSION="${TAG#v}"
BASE_VERSION="${VERSION%%-*}"

echo "=== Preflight for $TAG ==="

# 1. CRLF check
echo -n "CRLF in scripts... "
if grep -rPl '\r$' scripts/ gateway/*.sh 2>/dev/null; then
  echo "FAIL"; exit 1
fi
echo "ok"

# 2. Changelog
echo -n "CHANGELOG has ## [$BASE_VERSION]... "
grep -q "## \[${BASE_VERSION}\]" CHANGELOG.md || { echo "FAIL"; exit 1; }
echo "ok"

# 3. Version in meson.build
echo -n "meson.build version matches... "
grep -q "version: '${BASE_VERSION}'" meson.build || { echo "FAIL"; exit 1; }
echo "ok"

# 4. Compile
echo -n "C++ compile... "
meson compile -C builddir 2>&1 | tail -1
echo "ok"

# 5. Erlang
echo -n "Erlang compile + dialyzer... "
(cd gateway && rebar3 compile && rebar3 dialyzer) 2>&1 | tail -1
echo "ok"

# 6. Simulate changelog extraction
echo "=== Changelog preview ==="
awk "/^## \[${BASE_VERSION}\]/{found=1;next} /^## \[/{if(found) exit} found{print}" CHANGELOG.md

echo "=== Preflight PASSED for $TAG ==="

P5: sccache for C++ Compilation (2 hrs)

Install sccache on all runners. Configure: 

CC="sccache gcc-13"
CXX="sccache g++-13"
SCCACHE_DIR=/var/cache/sccache
SCCACHE_CACHE_SIZE=10G

On warm cache, a 3-file bug fix recompiles 3 files. Full rebuild: ~2 min instead of ~8 min.

P6: Local Artifact Cache on Self-Hosted Runners

Keep a local copy of every release artifact on the runner so that: 1. Re-runs don't need to rebuild just to re-upload 2. Artifacts survive GitHub's artifact storage cleanup (90-day expiry) 3. Slow uploads to GitHub don't block the pipeline — publish locally first, upload async

- name: Cache artifacts locally
  run: |
    LOCAL_CACHE="/var/cache/yuzu-releases/${GITHUB_REF_NAME}"
    mkdir -p "$LOCAL_CACHE"
    cp *.tar.gz *.deb *.rpm "$LOCAL_CACHE/" 2>/dev/null || true
    echo "Artifacts cached at $LOCAL_CACHE"

Create /var/cache/yuzu-releases/ with docker group ownership. Old releases can be pruned with a simple find -mtime +30 -delete cron.

P7: Workflow-Orchestrator as Release Guardian

The workflow-orchestrator agent should own the release process end-to-end:

  1. Pre-tag: Run preflight checks, validate runner health, simulate changelog extraction, verify .gitattributes
  2. Tag: Only create and push tag after all preflight checks pass
  3. Monitor: Watch the release workflow, surface failures immediately with root cause
  4. Post-release: Verify artifacts published, Docker images pullable, GitHub Release created

This replaces the current manual cycle of push-wait-read-fix-retag.

P8: Proactive Cache Pruning

GitHub Actions has a 10 GB cache limit per repo. Stale caches accumulate (52 caches, 10.4 GB at time of writing) and when the limit is hit, GitHub evicts LRU — which may be the warm vcpkg cache you actually need.

Add a post-job step or scheduled workflow that prunes stale caches: - Only the most recent cache per prefix matters (e.g. ccache-linux-gcc-13-*) - Delete all older entries for the same prefix after a successful build - Use gh api -X DELETE repos/{owner}/{repo}/actions/caches/{id}

Criteria for staleness: same prefix, older than current run's key. Prune immediately, don't wait for the 10 GB limit to bite.
