Plan: Fix Instruction Execution History — clickable rows + per-execution detail

Context

Today the Instructions → Executions tab in the dashboard renders a list of recent executions via GET /fragments/executions, but the rows are static text — there is no way to drill into one and see what happened: which agents ran it, who succeeded vs failed, and what each agent returned. The user's intent is to make this a real history table where clicking an execution reveals the historic results for that run.

The data is already captured by ExecutionTracker (executions + per-agent status) and ResponseStore (per-agent stdout/stderr/output), but the UI never joins them together.

Decisions confirmed with user

  • Detail UX: inline expanding drawer beneath the clicked row (matches the existing toggleDetail pattern in the results browser).
  • Scope of the tab: stays global by default; gains an optional ?definition_id=X filter so a future link from the Definitions list can scope it. One handler covers both modes.

What is currently in place

  • UI: server/core/src/instruction_ui.cpp:163-167 — Executions tab loads /fragments/executions on revealed.
  • List handler: server/core/src/workflow_routes.cpp:35-82 — emits a flat <table> with id (truncated), status, progress bar, dispatched_by, dispatched_at. No row click target. Ignores any definition_id query string even though ExecutionQuery::definition_id is supported.
  • Stores:
  • ExecutionTracker (execution_tracker.hpp) — query_executions(ExecutionQuery), get_execution(id), get_summary(id), get_agent_statuses(execution_id) are all available and tested.
  • ResponseStore (response_store.hpp) — query(instruction_id, ResponseQuery{since,until,agent_id,limit}) supports timestamp-window + agent filtering.
  • Linkage gap: responses has no execution_id column; we correlate via instruction_id = execution.definition_id plus the [dispatched_at, completed_at] timestamp window plus agent_id IN (per-agent statuses for this execution). Good enough for sequential dispatches of the same definition — non-invasive path. A schema migration to add execution_id is out of scope here and tracked as a follow-up.
  • Existing patterns to reuse:
  • Inline expanding-row drawer with toggleDetail(this) from results browser (dashboard_routes.cpp:1335-1357, dashboard_ui.cpp:602-616).
  • HTMX fragment route convention (/fragments/<entity> returning a partial).
  • Cisco Momentum tokens — --mds-color-state-hover, --mds-color-bg-success-tinted, --mds-color-bg-error-tinted, --mds-color-theme-text-tertiary, --mds-color-theme-indicator-error.
  • Existing CSS classes: .status-badge, .status-completed/.status-failed/.status-running/.status-pending, .progress-bar, .progress-fill.

Approach

Two server-side changes plus minimal HTML/JS wiring on the existing tab. No store / proto / migration changes.

1. Make the existing list richer and clickable

File: server/core/src/workflow_routes.cpp — handler at line 36.

  • Read optional definition_id query param; pass to ExecutionQuery so the tab can be filtered per-instruction without a second handler.
  • Render each row with:
  • Definition column (new): resolve e.definition_id → human name via instruction_store->get(definition_id) (already injected on this lambda's parameter list). Fall back to id-prefix if missing.
  • Time column: format dispatched_at as ISO-8601 UTC instead of raw epoch (use existing format_iso_utc helper if present in web_utils.hpp; otherwise add a small inline helper).
  • Click target: wrap row in <tr class="exec-row" onclick="toggleExecDetail(this)" hx-get="/fragments/executions/{id}/detail" hx-target="next .exec-detail-content" hx-trigger="click once" hx-swap="innerHTML"> so the detail loads lazily on first click and is just shown/hidden on subsequent clicks.
  • Detail row (new): empty <tr class="exec-detail"><td colspan="6"><div class="exec-detail-content"><div class="empty-state">Loading…</div></div></td></tr> beneath each row.

2. New detail fragment route

File: server/core/src/workflow_routes.cpp (add right after the list handler).

GET /fragments/executions/{id}/detail

Auth via existing auth_fn; require Read on the Execution securable (use the same permission predicate the existing list path uses — confirm during implementation). Return text/html containing three sections inside a <div class="exec-detail-grid">:

  1. Metadata block: definition name + id, status badge, dispatched_by, dispatched_at + completed_at (ISO-8601), parameters (pretty-printed JSON in <pre>), scope expression in <code>.
  2. Per-agent table (from execution_tracker->get_agent_statuses(id)): columns = agent_id (link to agent page if route exists), status badge, exit_code, dispatched_at→completed_at duration, error_detail (truncated, full text in title=).
  3. Responses table (from response_store->query(execution.definition_id, ResponseQuery{since=execution.dispatched_at, until=(execution.completed_at>0 ? completed_at : now()), limit=500}), then drop rows whose agent_id is not in the per-agent set): columns = agent_id, timestamp (ISO-8601), status, output (collapsed <details><summary> for long output), error_detail. Empty state if the response store is unavailable or the query returns 0 rows.

3. Tab wiring + JS

File: server/core/src/instruction_ui.cpp — same file as the existing tab markup.

  • Add a tiny toggleExecDetail(row) helper modeled on the existing toggleDetail(row) (search for it in the JS string in the same file or in dashboard_ui.cpp). It must collapse any other open detail row first so only one is expanded at a time.
  • Add CSS in the same <style> block (or wherever .status-badge lives): 
    .exec-row { cursor: pointer; }
.exec-row:hover { background: var(--mds-color-state-hover); }
.exec-row.expanded { background: var(--mds-color-state-selected); }
.exec-detail { display: none; }
.exec-detail.open { display: table-row; }
.exec-detail td { padding: 0.75rem; background: var(--mds-color-theme-bg-secondary, var(--bg)); }
.exec-detail-grid { display: grid; gap: 1rem; }
.exec-detail-grid h4 { margin: 0; font-size: 0.8rem; color: var(--mds-color-theme-text-tertiary); }

Out of scope (file as follow-ups, do not block this PR)

  • Adding execution_id column to responses for exact correlation. Today's timestamp+agent join is good enough for sequential dispatches of the same definition; concurrent dispatches of the same definition to overlapping agent sets will show the union of their responses in both detail views. File as a separate issue with the migration design.
  • Streaming live updates while an execution is running (would need SSE or HTMX poll). Today the detail snapshots at fragment-load time; user can re-click to refresh.
  • Pagination of the executions list beyond the existing limit=50. The current cap is fine for v1.
